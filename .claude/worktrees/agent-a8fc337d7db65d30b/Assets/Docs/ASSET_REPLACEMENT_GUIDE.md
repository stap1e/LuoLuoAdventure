# Asset Replacement Guide — LuoLuoTrip

## 1. Purpose

This document defines how temporary cylinder placeholder assets should be used and replaced in the LuoLuoTrip Unity project.

During prototype development, all missing 3D assets are represented by simple cylinder prefabs. These prefabs provide stable scene structure, colliders, faction identity, combat components, commander-control metadata, and debug readability before final art assets are available.

The goal is:

- Use simple placeholder geometry now.
- Keep gameplay logic stable.
- Replace only visual meshes later.
- Avoid breaking CharacterEntity, Combatant, AI, save data, faction data, or commander-control rules.

## 2. Placeholder Asset Location

All placeholder assets are stored under:

```text
Assets/Art/Placeholders/
  Materials/
  Prefabs/
```

Generated by menu item: **LuoLuoTrip/Setup/Generate Placeholder Assets**

## 3. Placeholder Prefab Naming

All placeholder prefabs use the `PH_` prefix.

Generated prefabs:

```text
PH_PlayerCommander_Cylinder.prefab
PH_MechaMinion_Cylinder.prefab
PH_BeastMinion_Cylinder.prefab
PH_CityLord_Cylinder.prefab
PH_WarKing_Cylinder.prefab
PH_Convoy_Cylinder.prefab
PH_EnergyNode_Cylinder.prefab
PH_ObjectiveMarker_Cylinder.prefab
```

Naming rules:

- `PH_` means placeholder.
- The middle name describes gameplay role.
- `_Cylinder` describes the temporary visual shape.
- Gameplay logic must live in components/data, not in the name.

## 4. Required Prefab Hierarchy

Every placeholder prefab follows this hierarchy:

```text
PrefabRoot
  Visual
    CylinderMesh
  Collision
    CapsuleCollider / BoxCollider
  Marker
```

Component placement on PrefabRoot:

```text
CharacterEntity
Combatant (added at scene instantiate time)
CombatController or SimpleCombatAI (added at scene instantiate time)
```

Important rule:

> Final art replacement should modify only the `Visual` child object unless a collider update is explicitly required.

Do not remove gameplay components from `PrefabRoot`.

## 5. Components That Must Not Be Deleted

When replacing a cylinder with a real model, do not delete or accidentally reset:

```text
CharacterEntity
Combatant
CombatController
SimpleCombatAI
Collider (CapsuleCollider / BoxCollider)
Rigidbody
CharacterData-bound fields (MainRace, SubFactionId, CharacterRole, CommandRank, etc.)
```

If a replacement model comes with its own Animator, MeshRenderer, or hierarchy, place it under `Visual` first and only then connect animation bridges if needed.

| Category | Material Name | Color |

| Mecha faction | MAT_PH_Mecha | Silver-gray |
| Beast faction | MAT_PH_Beast | Red-brown |

> Materials are only visual hints. Faction logic must come from CharacterEntity / CharacterData / SubFactionId, not material color.

## 8. Standard Replacement Workflow

To replace a placeholder cylinder with a final model:

1. Open the prefab in Prefab Mode.
2. Locate the `Visual` child.
3. Disable or remove the temporary cylinder mesh under `Visual`.
4. Drag the final model under `Visual`.
5. Reset local transform of the final model: Position 0,0,0; Rotation 0,0,0; Scale 1,1,1 then adjust.
6. Keep `PrefabRoot` unchanged.
7. Verify collider alignment.
8. If animation is available, connect Animator or AnimatorCombatBridge without deleting Combatant.
9. Save prefab.
10. Test in Play Mode.

## 9. Collider Replacement Rules

- Use simple colliders whenever possible.
- Prefer CapsuleCollider for character units.
- Prefer BoxCollider for convoy or objective assets.
- Avoid MeshCollider for moving combat units unless necessary.
- Keep collider size consistent with combat readability.

After collider adjustment, verify:

```text
Movement still works.
Attack hit detection still works.
Lock-on still works.
AI detection still works.
Mission triggers still work.
```

## 10. Animation Replacement Rules

When replacing static cylinders with animated models:

1. Keep gameplay root stable.
2. Put animated model under `Visual`.
3. Add Animator to the visual model if required.
4. Connect existing combat animation bridge if the project uses one.
5. Do not let animation root motion break gameplay movement unless intentionally supported.
6. Keep Combatant state machine as the source of combat truth.

```text
PrefabRoot = gameplay position and logic
Visual = animated mesh
Animator = visual presentation
Combatant = gameplay state
```

## 11. Verification Checklist After Replacement

### Scene-level checks
- The prefab appears in the scene.
- The prefab scale is readable.
- The unit does not fall through the floor.
- The collider matches the visible model.

### Combat checks
- The unit can take damage.
- The unit can attack if it should attack.
- The unit can be locked on.
- Hit feedback still plays.
- Dodge/movement still works if player-controlled.

### Commander checks
- ControlPermissionService returns the expected ControlMode.
- Low-rank units can be controlled by low-level commander.
- High-rank units reject or downgrade control as expected.
- SyncRate still appears in HUD.
- Denied reason still appears in HUD.

### Faction checks
- MainRace is correct.
- SubFactionId is correct.
- Dynamic hostility still works.
- AI target selection still uses faction logic, not material color.

### Mission checks
- Mission objective can still be completed.
- Convoy/resource/objective markers still trigger mission logic.
- Mission consequence still updates faction standing.
- Mission result still appears in UI.

### Save checks
- F5 save works.
- F9 load works.
- Commander profile is restored.
- Faction politics are restored.
- Character command fields are restored.

## 12. Common Mistakes

### Mistake 1: Replacing the whole prefab
Do not replace the whole prefab with a model import. Replace only the Visual child. Keep PrefabRoot logic components.

### Mistake 2: Losing CharacterEntity
If CharacterEntity is removed, faction, combat, save, and commander-control logic may break.

### Mistake 3: Using material color as faction logic
Materials are visual only. Faction logic must use MainRace and SubFactionId.

### Mistake 4: Putting gameplay scripts on imported model children
Imported model hierarchy may change after reimport. Keep gameplay scripts on stable project-owned prefab roots.

### Mistake 5: Animator root motion moving the logic root unexpectedly
If root motion causes incorrect movement, disable root motion or explicitly support it in the movement controller.

## 13. Recommended Final Asset Import Pattern

```text
Assets/Art/Characters/Mecha/
Assets/Art/Characters/Beast/
Assets/Art/Characters/Leaders/
Assets/Art/Environment/Objectives/
Assets/Art/Resources/
```

Final prefab structure should remain:

```text
GameplayPrefabRoot
  Visual
    FinalImportedModel
  Collision
  Marker
```

## 14. Replacement Priority

1. Player commander unit
2. Low-rank controllable mecha unit
3. Low-rank beast enemy unit
4. Convoy objective
5. Energy node
6. High-rank city lord
7. High-rank war king
8. Objective markers and environment props

Replace assets only after the corresponding gameplay logic is stable.

## 15. Final Rule

The art can change freely, but the gameplay root must stay stable.

Do not break:

```text
Commander authority
Faction politics
Mission consequence
Combat state
Save data
AI hostility
```

If a model replacement breaks gameplay, revert the visual child first before changing logic.
